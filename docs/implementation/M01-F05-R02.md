## Summary

An **Owner** can create **Manager** users for their organization, assigning each
Manager one or more stations. A created Manager gets **full access scoped to their
assigned station(s)** — there is no per-module permission matrix in this slice.
Onboarding is **phone + OTP self-set password**: the Owner submits the Manager's
name, +92 phone (optional email) and station assignment(s); the Manager receives
an SMS invite code and self-activates by setting their own password, after which
their next login shows **only** the assigned station(s). The Owner can toggle OTP
off per user (default on) — when off, the account is created already phone-confirmed
and a one-time temporary password is returned for the Owner to relay.

## Acceptance criteria (the test plan)

These become `[Fact]` names like `M01_F05_R02_OwnerCreatesManager` /
`M01_F09_R07_OtpNotRequiredReturnsTempPassword`:

- **AC1 (R02 create)** — Owner POSTs a valid create-Manager → `AppUser` created in
  the control plane with `OrganizationId = Owner.org` + Manager role, `UserStation`
  rows in the **tenant** DB, invite OTP issued + SMS sent; success response.
- **AC2 (tenant guard)** — Owner passes a `stationId` not in their org → failure,
  **no writes** in either context.
- **AC3 (R02 scoping)** — After activation, Manager login returns **only** the
  assigned station(s); Owner still sees all stations.
- **AC4 (R02 authz)** — A Manager (or any non-Owner) calls `POST /users/managers`
  → `403 Forbidden`, no implementation detail leaked.
- **AC5 (uniqueness)** — Duplicate phone → failure; duplicate email (when provided)
  → failure.
- **AC6 (R07 default)** — `requireOtp` omitted ⇒ treated as `true`; Manager created
  pending, login blocked until activated.
- **AC7 (R07 off)** — `requireOtp=false` ⇒ Manager created
  `PhoneNumberConfirmed=true`, temp password returned once, immediate login works.
- **AC8 (activation)** — Manager submits phone+code+newPassword to `/auth/activate`
  → password set, phone confirmed, login succeeds.

## Resolved decisions (settled during planning)

1. **Invite OTP purpose:** reuse existing `OtpPurpose.Signup` → **no Domain change,
   no migration**; `verify-phone` / `resend-otp` keep working unchanged.
2. **Activation UX:** add **one** new anonymous endpoint `POST /api/v1/auth/activate`
   (phone + code + newPassword) that verifies the invite OTP, sets the password, and
   sets `PhoneNumberConfirmed=true` in **one step / one code**. This is the clean
   realization of the chosen "phone + OTP self-set password" model (avoids the
   awkward two-code verify-phone-then-reset-password-otp path). Resend uses the
   existing `/auth/resend-otp`.
3. **OTP-not-required path (toggle off):** create the Manager
   `PhoneNumberConfirmed=true` with an **auto-generated temporary password returned
   once** in the response (Owner relays it). No forced-change enforcement this slice.
4. **Plan `MaxUsers` limit:** **enforced** (mirror the station-limit pattern); counts
   all active org users toward the cap (`-1` = unlimited).
5. **Manager list endpoint:** returns **Managers only** (Custom users are R03).

**Cross-context create-Manager choreography** (no distributed transaction;
`UnitOfWork` saves **tenant → control-plane**):

1. Authorize + resolve `orgId`/`role` from `ICurrentUserService`; recheck
   `role == owner` (mirror `CreateStationCommandHandler`). Fail closed otherwise.
2. **Validate stations FIRST** (before any write): load by ids via
   `IStationRepository`; assert all exist, `OrganizationId == orgId`, active. This
   read also pins the tenant context. Foreign/missing/inactive → fail, **nothing
   written** *(AC2 guard)*.
3. Plan `MaxUsers` check (count active org users; `-1` = unlimited).
4. Phone-uniqueness (query `UserManager.Users`); email-uniqueness when provided.
5. Create `AppUser` (`UserName = Phone`, `PhoneNumber = Phone`, `Email`, `FullName`,
   **`OrganizationId = orgId`** — critical for tenant routing on the Manager's later
   login). OTP-required ⇒ `CreateAsync(user)` with **no password** +
   `PhoneNumberConfirmed=false`. Toggle-off ⇒ generate temp password,
   `CreateAsync(user, tempPassword)`, `PhoneNumberConfirmed=true`. `CreateAsync` fail
   → return (nothing else written).
6. `AddToRoleAsync(Manager)`. Fail → compensate `DeleteAsync(user)`, return failure.
7. Stage `UserStation` rows (tenant) via `AddRangeAsync`.
8. OTP-required only: stage `PhoneVerification` (`Purpose = Signup`) exactly like
   `RegisterCommandHandler`.
9. `await _unitOfWork.SaveChangesAsync()` (tenant → control-plane). On throw →
   compensate: `DeleteAsync(user)` + best-effort `RemoveByUserAsync` for the tenant
   rows; log; return failure.
10. OTP-required: send invite SMS (fire-and-forget, try/catch — do **not** fail the
    command on SMS error; Owner can trigger resend).
11. Return `CreateManagerResponse` (`TemporaryPassword` only on the toggle-off path).

> *Why safe:* all validation (1–4) precedes all writes (5–9); the only multi-write
> failure window (9) has a single deterministic compensation (delete the just-created
> `AppUser`), and the `AppUser` is useless to login until OTP + stations exist.

## Implementation notes

Decisions and deviations during the build:

- **Solution format:** the solution is `server/FuelFlow.slnx` (XML format), not a `.sln`.
- **No migration shipped.** The optional `UserStation(UserId, StationId)` unique index was
  **skipped** (YAGNI): this slice only ever assigns distinct stations once at creation, and
  phone-uniqueness already blocks re-creating the same Manager. Zero control-plane and zero
  tenant migrations — so `## Layers touched → Infrastructure — EF config + migration` stays
  unchecked. Add the index if/when an "edit Manager stations" feature lands.
- **Invite OTP reuses `OtpPurpose.Signup`** (no new enum value, no migration). The new
  `POST /auth/activate` consumes the same Signup OTP and sets the first password in one step;
  resend is the existing `POST /auth/resend-otp`.
- **Password set on activation** uses the Identity reset-token path
  (`GeneratePasswordResetTokenAsync` + `ResetPasswordAsync`) uniformly — it works whether or
  not a password already exists, so a separate `AddPasswordAsync` branch was unnecessary
  (minor deviation from the Phase 2 note).
- **Temp password (toggle-off path)** is a 16-char random string guaranteed to include
  upper/lower/digit/symbol, returned once in `CreateManagerResponse.TemporaryPassword`.
- **`GetManagers`** uses `UserManager.GetUsersInRoleAsync("Manager")` filtered to the org,
  then one `GetStationIdsByUserIdAsync` per manager (N+1, acceptable for the low manager
  count per org; revisit with a bulk query if needed).
- **Backend xUnit tests deferred** — no test project in the repo (see Verification note).
- **`dotnet build` caveat:** a full `FuelFlow.slnx` build fails the *dependency-copy* step
  while `scripts/dev.ps1` holds `FuelFlow.Api/bin` DLLs — not a compile error. Verified
  clean compile via `dotnet build FuelFlow.Infrastructure` (0 errors).
- **Pre-existing role-casing bug fixed here (E2E-surfaced).** Backend auth responses serialized
  roles lowercase (`Roles = userRoles.Select(r => r.ToLower())` in `Login`/`GetCurrentUser`/
  `RefreshToken`) but `lib/roles.ts` gates case-sensitively on `"Owner"` — so all role-gated nav +
  route guards failed (M07-F07 contract). Fixed both layers (user chose "both"): keep enum casing
  in the three handlers + make `hasAnyRole` case-insensitive. The running dev API still serves
  lowercase until restarted; the case-insensitive frontend tolerates either, so no restart is
  required for correctness.
- **Discovered, not fixed — M14 login 500 (deferred per user; needs a separate M14 item).**
  A fresh (cookieless) login of an **onboarded** user returns **500**: `LoginCommandHandler` (line
  ~123) loads org/stations via the tenant-routed `OrganizationRepository`, whose
  `TenantDbContextAccessor` resolves the tenant DB **only from the JWT `org_id` claim** — which does
  not exist during the pre-authentication login request, so it throws
  `"GetContextAsync called for a request with no org_id claim"`. Masked in normal use because the
  browser keeps the `org_id` cookie after the post-onboarding token refresh; a cookie-cleared login
  exposes it. Pre-existing M14, **out of M01-F05-R02 scope**, but severe (blocks fresh login for any
  onboarded user). Proper fix: let login resolve the tenant from `user.OrganizationId` (e.g. an
  explicit org override on the resolver/accessor) rather than the absent claim. **E2E walked around
  it** by registering a fresh Owner (whose pre-org first login succeeds, and onboarding re-issues the
  cookie with `org_id`).
