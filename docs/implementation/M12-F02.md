## Summary

A development-environment-only flag that relaxes the dashboard route guard from [M12-F01-R16](../MODULES.md#m12-f01--onboarding-wizard). Backend computes a `devBypassActive` boolean as `IHostEnvironment.IsDevelopment() && Features:OnboardingDevBypass == true`; the boolean rides on the existing `AuthResponse` (login, refresh, `GET /auth/me`). Frontend consumes it to (a) relax the `/dashboard` `beforeLoad` guard, (b) show a "Skip to Dashboard (dev only)" affordance in the wizard chrome, and (c) render a persistent dashboard banner when bypass is active and onboarding is incomplete. Wizard intra-step validation and backend endpoint preconditions are **not** relaxed — the bypass is purely an entry-gate concern.

## Acceptance criteria (the test plan)

From [`MODULES.md` M12-F02](../MODULES.md#m12-f02--onboarding-dev-bypass):

- **AC1** Given `ASPNETCORE_ENVIRONMENT=Production`, When `Features:OnboardingDevBypass=true` is set in env, Then `GET /auth/me` returns `devBypassActive: false`. _(Production-safety property — the C#-level `IsDevelopment()` check short-circuits even if config says otherwise.)_
- **AC2** Given `ASPNETCORE_ENVIRONMENT=Development` AND `Features:OnboardingDevBypass=true`, When an authenticated user calls `GET /auth/me`, Then the response includes `devBypassActive: true`.
- **AC3** Given `devBypassActive=true` AND `isSetupComplete=false`, When the user navigates to `/dashboard`, Then the dashboard renders without redirect.
- **AC4** Given `devBypassActive=true`, When the user is on any wizard step, Then a "Skip to Dashboard (dev only)" button is visible between the progress bar and step header; clicking it navigates to `/dashboard`.
- **AC5** Given `devBypassActive=true` AND `isSetupComplete=false`, When the dashboard renders, Then a banner "Dev bypass active — onboarding incomplete" is visible at the top.
- **AC6** Given `devBypassActive=true` AND `isSetupComplete=true`, When the dashboard renders, Then no bypass banner is shown.
