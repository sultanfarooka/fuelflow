## Summary

Post-onboarding self-service management of a station's fuel types for Owner +
Manager, rendered inside the Fuel Types tab of Station Configuration. Users can
view all fuel types for the station (with source + status + reference counts),
add a custom type or one from the OMC catalog, rename, and activate/deactivate
types — capabilities previously only available inside the onboarding wizard
(M12-F01). Supersedes `M02-F01-R03` ("not a self-service feature").

## Key model facts (from code review during planning)

- `FuelType` is a **control-plane** entity (`ControlPlaneDbContext`). A station's
  fuel types are **per-station rows** (`StationId == stationId`): OMC-selected
  fuels are materialized at onboarding with `IsCustom=false` + `OMCId` set;
  custom fuels have `IsCustom=true`. `GetAllForStationAsync` filters on
  `StationId` only — there is **no shared/global per-station merge**, so a single
  `IsActive` column covers both OMC-derived and custom rows. **No override
  table is needed.**
- Tanks (`FuelTank.FuelTypeId`) and prices (`FuelPrices.FuelTypeId`) reference
  the per-station `FuelType.Id`. Nozzles reference tanks, not fuel types.
- Existing handlers: `GetFuelTypesByStationQuery`, `CreateFuelTypeCommand`,
  `DeleteFuelTypeCommand`; repo `IFuelTypeRepository`
  (`GetByIdAsync`, `GetAllForStationAsync`, `AddAsync`, `DeleteAsync`).

## Decisions (from the planning interview)

1. **Soft-deactivate only on the management surface.** Add an `IsActive` flag;
   the management UI exposes activate/deactivate, never hard delete. The
   existing hard `DELETE` endpoint is **retained untouched** for the onboarding
   wizard's "remove a just-added fuel" flow (M12-F01, out of scope here).
2. **Audit via Serilog now.** Log add/rename/activate/deactivate with actor +
   station + before/after through the existing `ILogger` pipeline. Swap to the
   real `AuditLog` table when `M01-F08` ships (R07 is satisfied, not deferred).
3. **Activate/deactivate AND rename apply to both OMC-derived and custom types**
   — they are per-station rows, so both are safe to mutate per station. This
   **refines `M02-F01-R03` and `M08-F08-R03`**: the "OMC types are read-only"
   clause is dropped (it assumed shared catalog rows, which is not the model).
4. **List shows tank count + whether an active price exists** (not nozzle
   counts). These two drive the R06 "not yet sellable" flag.

## Acceptance criteria (the test plan)

From `MODULES.md` (AC1–AC6). Test names like
`M08_F08_R05_DeactivateBlockedWhileReferenced`:

- **AC1** — Owner on the Fuel Types tab sees all configured types with name,
  unit, source (OMC/Custom), active/inactive status, and references (tank count +
  active-price indicator).
- **AC2** — Owner adds a custom type (unique name + unit) → appears active and
  flagged "not yet sellable" until it has a price and a tank.
- **AC3** — Manager adds a type whose name duplicates an existing one for the
  station → blocked by inline (Zod) + API (FluentValidation/handler) error.
- **AC4** — Owner deactivates a type with no active references → it disappears
  from new price/tank pickers but remains in historical reports.
- **AC5** — Owner tries to deactivate a type referenced by an active price or a
  tank → `409` lists the blocking references; the type stays active.
- **AC6** — Any add/rename/activate/deactivate writes a Serilog audit entry with
  actor, station, before/after.

## Implementation notes

- **Phase 2 — reference counts computed in-memory.** Rather than adding
  `CountByFuelTypeAsync` / `HasActivePriceAsync` repo methods, the list handler
  loads the station's tanks and prices once (existing list methods) and computes
  `TankCount` + `HasActivePrice` in memory. Per-station data volume is small;
  this avoids N+1 queries and keeps the repo interfaces unchanged. "Active price"
  reuses the existing effective-window rule (`EffectiveFrom <= now && (EffectiveTo
  == null || EffectiveTo > now)`).
