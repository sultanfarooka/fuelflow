## Summary

Post-onboarding self-service management of a station's fuel tanks for Owner + Manager, rendered as the third child of the Station Configuration hub. Replaces the `<UnderDevelopment />` placeholder. Users see a list of tanks at the station with name + capacity + fuel type + dip-chart status + nozzle count, can **add** a tank (capacity + fuel-type required, name optional), **edit** any of those fields (with extra confirmation when reassigning the fuel type ‚Äî M02-F03-R06), and **delete** with a structured 409 reference guard listing blocking nozzles / readings. Implements R01 (CRUD UI) and confirms R02 (uniqueness). **Out of scope:** dip-chart upload (M08-F04), stock-level / %-full visualization (M02-F03-R05), inventory drill-down (M02).

## Decisions (resolved during planning interview)

1. **Delete UX: hard-delete + structured 409 reference guard.** Tanks are physical infrastructure that get *decommissioned*, not *deactivated*. The handler is upgraded to preflight nozzle count + tank-reading count and return `Result.Failure(...)` whose controller layer maps to a 409 `{ error, references }` payload (same shape as M08-F08's deactivate 409). The Delete confirm dialog renders the references list inline when present. **No** `IsActive` flag on `FuelTank` (no migration).
2. **Fuel-type reassignment is allowed on Edit, with extra confirmation** (M02-F03-R06). The Edit dialog exposes Name + Capacity + Fuel Type. When Fuel Type changes from its original, render a destructive-tone warning ("Changing fuel type ‚Äî historical readings + nozzle assignments will keep their old fuel type for reports. Confirm carefully.") and require a second click on Save. Backend already accepts the change; no extra validation.
3. **Nozzle count is added to `FuelTankDto`.** `GetFuelTanksByStationQueryHandler` joins with `FuelNozzles` (same one-pass in-memory pattern M08-F08 uses for tank counts). New column on the table; populates the 409 reference message verbatim. Small backend tweak.
4. **Serilog audit added to all three handlers** (`Create`, `Update`, `Delete`) matching the M08-F08 + M06-F01 line shape. Persistent `AuditLog` table rides with M01-F08 ‚Äî pattern stays consistent.
5. **Visual chrome inherits from M08-F08 + M06-F01 by reuse** (see "Visual consistency" section below). Third adopter of `<ConfigPanelCard>` + shared `<DataTable>` + extracted badges; no new shell extractions required.
6. **Capacity input quirk** ‚Äî `step="any"` plus a separate "Liters" suffix; precision is whole-litre in practice but `decimal` on the entity. Frontend validator: `capacityLiters > 0`, `<= 1,000,000` sanity cap (a real fuel tank is 5k‚Äì200k L; cap leaves 5√ó headroom).

## Implementation notes

- **Open-question defaults applied without prompt** (Auto Mode): capacity downward edits with stored readings ‚Äî allowed (Q1, yes); Add-tank button hidden + empty-state CTA when station has no fuel types (Q2, yes); `HasDipChart` plumbing confirmed via `GetFuelTanksByStationQueryHandler` reading `t.DipChart != null` from EF eager-load (Q3, no plumbing change needed).
- **Phase 1 ‚Äî preflight only checks nozzles, not readings.** `FuelTankReadings` has an EF config + entity but no `IFuelTankReadingRepository` on this branch. Adding one is M02 (Fuel Inventory) territory; doing it here would be scope creep. Net effect: the structured 409 message lists "N nozzles" when applicable; if a tank has stored readings but no nozzles (a degenerate state), the EF cascade-violation falls through to the generic 400 fallback. Acceptable for v1 because in normal use, a tank with readings always has at least one nozzle that recorded them.
- **Phase 1 ‚Äî `DeleteFuelTankCommand` return type changed from `Result<bool>` to `Result<DeleteFuelTankResponse>`.** Carries the structured `Blocked` + `BlockingReferences` payload the controller maps to 409. The bool was carrying no information beyond the `Result.IsSuccess` flag anyway.
- **Post-review scope expansion (dip-chart UX).** User requested dip-chart support directly on the Fuel Tanks panel rather than waiting for a standalone M08-F04 PR: optional CSV upload while adding a tank, and a per-tank "Dip chart" action that views + replaces. Backend was already complete (`UploadDipChartCommand` + `GetDipChartByTankQuery`; upload handler is upsert ‚Äî deletes the prior chart before insert), so no schema or handler changes. Frontend changes:
  - Extracted `parseDipChartCsv` to `lib/parse/dip-chart-csv.ts` (was duplicated across `step3-tanks.tsx` + `step4-dip-charts.tsx`). Panel is the third call site.
  - `TankFormDialog` Add mode: optional file-input below Fuel Type, parses inline, shows entry count + parse errors. On submit, `createMutation` chains the tank create with a follow-up `uploadDipChart` if entries were supplied. If the tank create succeeds but the dip-chart upload fails, surfaces a `toast.warning` with the partial-success message so the user can retry from the row's Dip chart action.
  - New `DipChartDialog` co-located in the panel file. Lazy-loads the current chart (separate query key `DIP_CHART_KEY`). Shows a compact entries table with "Show all" toggle when >8 entries. CSV file input + parsed-preview; Save calls `uploadDipChart` (upsert). On success: invalidates the per-tank dip-chart query AND the tanks list query so the `hasDipChart` chip flips.
  - Third action button "Dip chart" added to row + mobile card.
