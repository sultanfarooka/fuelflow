## Summary

Post-onboarding self-service management of a station's fuel tanks for Owner + Manager, rendered as the third child of the Station Configuration hub. Replaces the `<UnderDevelopment />` placeholder. Users see a list of tanks at the station with name + capacity + fuel type + dip-chart status + nozzle count, can **add** a tank (capacity + fuel-type required, name optional), **edit** any of those fields (with extra confirmation when reassigning the fuel type — M02-F03-R06), and **delete** with a structured 409 reference guard listing blocking nozzles / readings. Implements R01 (CRUD UI) and confirms R02 (uniqueness). **Out of scope:** dip-chart upload (M08-F04), stock-level / %-full visualization (M02-F03-R05), inventory drill-down (M02).

## Key model facts (from code review during planning)

- **`FuelTank` is per-tenant** (`AppDbContext`, table `fuel_tanks`). Columns: `Id`, `Name?` (nullable, ≤200 chars), `CapacityLiters` (decimal), `FuelTypeId` (cross-context plain Guid → control-plane `FuelType`), `StationId`. Per-station name uniqueness enforced in `CreateFuelTankCommandHandler`. R02 is **already Done**.
- **Backend CRUD is already wired end-to-end:**
  - Entity + EF config + per-tenant migration ✓
  - DTOs (`FuelTankDto`, `CreateFuelTankRequest`, `UpdateFuelTankRequest`, `CreateFuelTankResponse`) ✓
  - `CreateFuelTankRequestValidator` — `CapacityLiters > 0`, `FuelTypeId` not empty, `Name` ≤ 200. **No `UpdateFuelTankRequestValidator` yet** — this PR adds one.
  - Handlers: Create / Update / Delete / GetByStation ✓
  - Repo: `IFuelTankRepository` — `GetByIdAsync`, `GetAllByStationIdAsync`, `AddAsync`, `UpdateAsync`, `DeleteAsync`
  - Controller: `[Authorize(Roles="Owner,Manager")] [Route("api/v1/stations/{stationId:guid}/fuel-tanks")]` — GET / POST / PUT / DELETE
- **`FuelTankDto` already exposes** `HasDipChart` + `DipChartEntryCount`. Missing: `NozzleCount` — this PR extends the DTO + the query handler.
- **Delete handler is currently soft** — it catches the EF cascade-violation exception with a generic 400 message ("It may be referenced by nozzles or other records."). This PR upgrades it to **preflight-check** references and return a structured **409** with the blocking list (mirrors M08-F08's deactivate guard pattern).
- **`SetFuelPriceCommandHandler` has Serilog audit; `CreateFuelTypeCommandHandler` has Serilog audit; the three tank handlers do not.** This PR adds matching `AUDIT FuelTank.Create / Update / Delete` lines.
- Frontend `lib/api/stations/fuel-tanks.ts` is already typed with `FuelTankDto`, `CreateFuelTankRequest`, `UpdateFuelTankRequest`, `createFuelTank`, `updateFuelTank`, `getFuelTanksByStation`, `deleteFuelTank`. Will likely need a small extension for the structured 409 references payload.
- Setup-wizard precedent: `components/station-setup/step3-tanks.tsx` (per-fuel-type add tank form) — borrow form layout patterns but not the per-fuel-type grouping (the management UI is a flat tank-centric list).

## Decisions (resolved during planning interview)

1. **Delete UX: hard-delete + structured 409 reference guard.** Tanks are physical infrastructure that get *decommissioned*, not *deactivated*. The handler is upgraded to preflight nozzle count + tank-reading count and return `Result.Failure(...)` whose controller layer maps to a 409 `{ error, references }` payload (same shape as M08-F08's deactivate 409). The Delete confirm dialog renders the references list inline when present. **No** `IsActive` flag on `FuelTank` (no migration).
2. **Fuel-type reassignment is allowed on Edit, with extra confirmation** (M02-F03-R06). The Edit dialog exposes Name + Capacity + Fuel Type. When Fuel Type changes from its original, render a destructive-tone warning ("Changing fuel type — historical readings + nozzle assignments will keep their old fuel type for reports. Confirm carefully.") and require a second click on Save. Backend already accepts the change; no extra validation.
3. **Nozzle count is added to `FuelTankDto`.** `GetFuelTanksByStationQueryHandler` joins with `FuelNozzles` (same one-pass in-memory pattern M08-F08 uses for tank counts). New column on the table; populates the 409 reference message verbatim. Small backend tweak.
4. **Serilog audit added to all three handlers** (`Create`, `Update`, `Delete`) matching the M08-F08 + M06-F01 line shape. Persistent `AuditLog` table rides with M01-F08 — pattern stays consistent.
5. **Visual chrome inherits from M08-F08 + M06-F01 by reuse** (see "Visual consistency" section below). Third adopter of `<ConfigPanelCard>` + shared `<DataTable>` + extracted badges; no new shell extractions required.
6. **Capacity input quirk** — `step="any"` plus a separate "Liters" suffix; precision is whole-litre in practice but `decimal` on the entity. Frontend validator: `capacityLiters > 0`, `<= 1,000,000` sanity cap (a real fuel tank is 5k–200k L; cap leaves 5× headroom).

## Implementation notes

- **Open-question defaults applied without prompt** (Auto Mode): capacity downward edits with stored readings — allowed (Q1, yes); Add-tank button hidden + empty-state CTA when station has no fuel types (Q2, yes); `HasDipChart` plumbing confirmed via `GetFuelTanksByStationQueryHandler` reading `t.DipChart != null` from EF eager-load (Q3, no plumbing change needed).
- **Phase 1 — preflight only checks nozzles, not readings.** `FuelTankReadings` has an EF config + entity but no `IFuelTankReadingRepository` on this branch. Adding one is M02 (Fuel Inventory) territory; doing it here would be scope creep. Net effect: the structured 409 message lists "N nozzles" when applicable; if a tank has stored readings but no nozzles (a degenerate state), the EF cascade-violation falls through to the generic 400 fallback. Acceptable for v1 because in normal use, a tank with readings always has at least one nozzle that recorded them.
- **Phase 1 — `DeleteFuelTankCommand` return type changed from `Result<bool>` to `Result<DeleteFuelTankResponse>`.** Carries the structured `Blocked` + `BlockingReferences` payload the controller maps to 409. The bool was carrying no information beyond the `Result.IsSuccess` flag anyway.
