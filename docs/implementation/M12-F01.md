## Summary

Replaces the current single-step `/onboarding` page with a 9-step, production-grade
wizard that takes a newly registered Owner from organization creation to a fully
operational station — covering fuel types, opening prices, underground tanks with
dip charts, nozzles, shift scheduling, payment methods, and optional extras (bank
account, first manager invite). All step UI is designed and built from scratch in
`components/onboarding/`; the existing prototype components in
`components/station-setup/` are **not** reused in the wizard. Data is saved
progressively after each step. Dashboard access is blocked via a route guard until
`Station.IsSetupComplete = true`. The `/dashboard/station/:stationId/setup` wizard
is retained unchanged for additional-station configuration.

## Wizard step map

| Step | Name | Required | API call on Next / Finish |
|---|---|---|---|
| 1 | Org + Station | Yes | `POST /onboarding` (existing) |
| 2 | Fuel types | Yes | `POST /fuel-types` per type (existing) |
| 3 | Opening prices | Yes | `PUT /fuel-prices` per type (existing) |
| 4 | Tanks + dip charts | Yes | `POST /fuel-tanks` + `POST /fuel-tanks/{id}/dip-chart` per tank (existing) |
| 5 | Nozzles | Yes | `POST /nozzles` per nozzle (existing) |
| 6 | Shifts + Payment methods | Yes | `POST /stations/{id}/shift-config` (new) + `PUT /stations/{id}/payment-methods` (new) |
| 7 | Bank account | Skippable | `POST /organizations/{id}/bank-accounts` (new) |
| 8 | Invite first manager | Skippable | `POST /organizations/{id}/users` (depends on M01-F05-R02) |
| 9 | Summary + Finish | Yes | `POST /stations/{id}/complete-setup` (new) |

## Acceptance criteria (the test plan)

- **AC1** Given a newly registered user with no organization, When they log in, Then they are redirected to `/onboarding` Step 1.
- **AC2** Given a user who completed Steps 1–3 and dropped off, When they log in again, Then the wizard auto-advances to Step 4.
- **AC3** Given the user clicks Next on Step 2 with zero fuel types, When validated, Then an inline error blocks advancement.
- **AC4** Given the user navigates directly to `/dashboard` while `isSetupComplete = false`, When the route guard evaluates, Then they are redirected to `/onboarding`.
- **AC5** Given all required steps complete, When the user clicks "Finish setup" on Step 9, Then `POST /complete-setup` succeeds, `isSetupComplete = true`, and the user lands on `/dashboard`.
- **AC6** Given `POST /complete-setup` is called with a missing fuel price, When the handler validates, Then it returns `400` naming the unpriced fuel type.
- **AC7** Given the wizard on viewport < 640px, When any step renders, Then no horizontal scroll occurs and all controls are reachable.
- **AC8** Given Step 6 with no shift names entered, When Next is clicked, Then an inline error blocks advancement.
- **AC9** Given Step 7 (bank account), When "Skip for now" is clicked, Then wizard advances without an API call.
- **AC10** Given Step 8 (invite manager), When "Skip for now" is clicked, Then wizard advances to Step 9 without an API call.

### Phase 4 — API: 4 new endpoints

All endpoints require `[Authorize(Roles = "Owner")]` and a multi-tenancy guard that verifies the `stationId`/`orgId` belongs to the caller's organization.

- [ ] `POST /api/v1/stations/{stationId}/shift-config` → dispatch `CreateShiftConfigCommand`; return 200 + `ShiftConfigDto`.
- [ ] `GET  /api/v1/stations/{stationId}/shift-config` → dispatch `GetShiftConfigQuery`; return 200 + `ShiftConfigDto` or 404.
- [ ] `PUT  /api/v1/stations/{stationId}/payment-methods` → dispatch `UpdatePaymentMethodsCommand`; return 200 + updated `StationDto`.
- [ ] `POST /api/v1/organizations/{orgId}/bank-accounts` → dispatch `CreateBankAccountCommand`; return 201 + `BankAccountDto`.
- [ ] `GET  /api/v1/organizations/{orgId}/bank-accounts` → dispatch `GetBankAccountsQuery`; return 200 + list.
- [ ] `POST /api/v1/stations/{stationId}/complete-setup` → dispatch `CompleteStationSetupCommand`; if success return 200 `{ success: true }`; if unmet return 400 `{ success: false, unmetConditions: string[] }`.
- [ ] *Acceptance: All 6 endpoints appear in Swagger with correct routes and auth.*

### Phase 6 — Frontend: wizard shell

Rewrite `fuel-flow-web/src/routes/onboarding/index.tsx` as the wizard orchestrator.
All visual design is new; do not reference or import from `components/station-setup/`.

**Wizard state (local, not persisted):**
```
currentStep: 1–9
stationId: string | null      // set after Step 1 succeeds
isResuming: boolean           // true while computing resume step on mount
```

**Resume detection (`computeResumeStep`):**

Async helper called on mount when `organization` already exists in auth store (i.e., the user dropped off after Step 1). Queries in parallel:
- `getFuelTypesByStation(stationId)` → step 2 done if count > 0
- `getFuelPricesByStation(stationId)` → step 3 done if every fuel type has a price
- `getFuelTanksByStation(stationId)` → step 4 done if every fuel type covered with dip chart
- `getFuelNozzlesByStation(stationId)` → step 5 done if count > 0
- `getShiftConfig(stationId)` → step 6 done if response is non-null
- Returns first step number whose condition is not met (2–7); defaults to 7 if all required steps are done (lands on first optional step, bank account).

**Shell tasks:**

- [ ] Implement wizard shell with `currentStep` state, `stationId` state, and `isResuming` state.
- [ ] On mount: if `useAuthStore().organization` exists, run `computeResumeStep` and jump to the returned step.
- [ ] While `isResuming`, render a centered loading spinner with "Resuming your setup…" text.
- [ ] Render a **progress bar** at the top: 9 labeled segments; completed segments filled with `bg-primary`, active segment pulsing, future segments `bg-muted`.  On mobile collapse labels and show "Step X of 9" text only.
- [ ] Render a **step header** block below the progress bar: step number chip, step title, one-line description. Changes per step.
- [ ] Render the active step component below, passing `stationId`, `onNext(stationId?)`, `onBack()`, `onSkip()` callbacks.
- [ ] "Back" is visible from Step 2 onward (Step 1 has no Back). "Skip for now" is visible only on Steps 7 and 8.
- [ ] *Acceptance: Navigate to `/onboarding` as a new user → Step 1 renders with no console errors. Complete Step 1 → progress bar advances, Step 2 header appears.*

## Open questions

- **Auth store refresh after Step 9:** The cleanest way to get `isSetupComplete = true` into the store after `complete-setup` succeeds is to re-fetch `/auth/me` (the "me" endpoint returns a `StationDto` array including the flag). Alternatively the `complete-setup` response can include the updated `StationDto`. Decide in Phase 3 based on the handler implementation.
- **`AcceptedPaymentMethods` storage:** JSONB array on `Station` vs a separate `StationPaymentMethod` join table. JSONB is simpler for read and the list is small (≤5 items); use JSONB unless the team prefers relational.
- **Shift3 nullable fields:** Some ORMs require explicit nullable handling for `TimeSpan?`. Confirm EF Core maps `TimeSpan?` → nullable `time` correctly for PostgreSQL; if not, store as `int? Shift3StartMinutes` and convert in the DTO mapper.
- **Manager invite Step 8 gate:** Determine before Phase 8 whether M01-F05-R02 will ship in the same PR or after. If after, the step renders disabled UI only; if in the same PR, the invite call must be fully wired.
