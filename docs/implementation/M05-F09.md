## Summary

Provides a per-organization registry of income and expense account heads (Pakistani accounting term: *khatas*). Default expense heads are seeded when an organization is first created (onboarding Step 1); fuel-type income heads (`Fuel Sales PMG (Cash/Card)`, `Credit Sales PMG`, etc.) are seeded when fuel types are selected (onboarding Step 2). Owners and Managers can add custom heads (e.g., "Tyre Shop Rent") and rename or deactivate any non-system head. Account heads are consumed by **M05-F03** (daily expenses) and **M05-F10** (other income) as their categorization layer and by **M07-F03** (P&L report) for roll-up.

This PR covers **backend only**. Frontend account-head management UI ships in a follow-up PR alongside M05-F03 or M05-F10.

## Acceptance criteria (the test plan)

- **AC1** Given a new organization completes onboarding with PMG and HSD active, When seeding completes, Then four system income heads exist: `Fuel Sales PMG (Cash/Card)`, `Credit Sales PMG`, `Fuel Sales HSD (Cash/Card)`, `Credit Sales HSD` — and seven default expense heads are present.
- **AC2** Given an expense head with 0 existing transactions, When deactivation is requested, Then it succeeds and `IsActive` becomes `false`.
- **AC3** Given a system-managed head (`IsSystemManaged = true`), When deactivation is attempted, Then `400 Bad Request` is returned.
- **AC4** Given a custom INCOME head "Tyre Shop Rent" is created, When the list is queried with `?type=Income`, Then it appears in the result.
- **AC5** Given two custom heads with the same name in the same organization, When the second is created, Then `409 Conflict` is returned.

> Note: AC2's "0 transactions" check is always true in this PR because M05-F03 and M05-F10 are not yet built. The `IAccountHeadUsageChecker` service is implemented as a stub that returns `false`; it is extended when those features ship.

## Open questions

- **Fuel type name format in income head names**: the seeder uses the `FuelType.Name` string (e.g., `"PMG"`, `"HSD"`, `"HOBC"`). If a user adds a custom fuel type (M12-F01-R04 allows this), the seeder should use whatever name they entered. Verify that the fuel type name field in the domain is already accessible from the onboarding Step 2 handler. If the custom fuel type name is very long, the 100-char `AccountHead.Name` limit could be hit — add a truncation safeguard in the seeder if needed.

- **Re-seeding on fuel type changes**: if an Owner later adds a new fuel type to an existing station (outside onboarding), should new income heads be seeded? This PR only hooks into onboarding. Handling post-onboarding fuel-type additions is out of scope here — flag as a follow-up when the "add fuel type" post-onboarding flow is built.
