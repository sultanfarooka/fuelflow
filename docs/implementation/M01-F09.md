## Summary

Phone (+92) becomes the primary identifier across registration, login,
verification, and recovery for Pakistani station owners who often lack stable
email accounts. Email is optional and acts as a verified fallback. The PR
also delivers a phone-change flow (R11), per-phone + per-IP OTP rate limiting
(R12), and lights up SMS-OTP password recovery (`M01-F04-R04`). SMS is sent
via a self-hosted [`capcom6/sms-gateway`](https://github.com/capcom6/android-sms-gateway)
running in Docker (private mode).

## Key technical decisions

- **SMS provider:** self-hosted `capcom6/sms-gateway` (private mode).
  Endpoint `POST {BaseUrl}/3rdparty/v1/message`, HTTP Basic auth, body
  `{ "textMessage": { "text": "…" }, "phoneNumbers": ["+92…"] }`. Requires
  **MariaDB** + **Firebase Cloud Messaging (FCM)** credentials.
- **OTP storage:** dedicated `PhoneVerification` table (audit-friendly,
  supports R04 limits and R12 daily cap via index).
- **OTP hashing:** **HMAC-SHA256 with server-side pepper** (correct primitive
  for short-lived 6-digit codes; BCrypt would be overkill). Pepper in
  user-secrets / env.
- **Login UX:** single combined "Phone or email" input (Zod union validates).
- **Daily OTP cap:** configurable via `Otp:DailyCapPerPhone` (default `10`).
- **i18n:** full **en + ur** in this PR, including retro-i18n of touched
  email-era screens (login, register, verify-email, forgot, reset).
