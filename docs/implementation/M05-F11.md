## Summary

Single `FinancialEntries` table that every financial event in the system writes
to. Every row is one monetary event with a mandatory amount, date, and entry
type. Optional dimension columns (`AccountHeadId`, `StationId`, `BankAccountId`,
`CustomerId`, `SupplierId`, etc.) are populated only when relevant. Features
such as Cash Book (F05) and Bank Account balance (F04) are **derived views**
over this table, not separate stores.

This PR covers **backend only** — Domain entity + enums, EF configuration +
migration, repository, a read-only list/filter query endpoint, and a
ManualAdjustment endpoint for corrections (R05). No public "create entry"
endpoint — consuming features (M05-F03 expenses, M05-F10 other income,
M04-F06 shift close, etc.) create entries through their own commands using
the repository directly.

Also updates `IAccountHeadUsageChecker` from its stub to a real implementation
that queries `FinancialEntries` for rows referencing a given `AccountHeadId` —
completing the deferred check from M05-F09-R05.

- [ ] `GetFinancialEntriesQueryHandler`

  - Calls `IFinancialEntryRepository.GetListAsync` with all filters.

- [ ] `AdjustFinancialEntryCommandHandler`
  - Gets `organizationId` from `ICurrentUserService`.
  - Loads original entry via `IFinancialEntryRepository.GetByIdAsync`.
  - Returns failure if not found (→ 404).
  - Returns failure if `IsSystemGenerated = true` (→ 400 "System-generated
    entries cannot be adjusted").
  - Creates **two** new entries sharing a new `TransactionGroupId`:
    1. **Reversal row:** same dimensions as original, `Amount` negated,
       `EntryType = ManualAdjustment`, `AdjustmentReason` = request reason,
       `IsSystemGenerated = true`.
    2. **Correction row:** same dimensions as original (overridden by request
       fields where provided), `Amount = CorrectedAmount`,
       `EntryType = ManualAdjustment`, `AdjustmentReason` = request reason.
  - Calls `AddRangeAsync` + `SaveChangesAsync`.
  - Returns the pair as DTOs.

## Open questions

- **Correction of ManualAdjustment entries** — can an adjustment itself be
  adjusted? Current design says yes (it's not `IsSystemGenerated`), creating
  another reversal+correction pair. This may produce long chains. For v1 this
  is acceptable; revisit if it causes confusion.

- **Audit trail integration (M01-F08)** — R05 says adjustments should be logged
  to the audit trail. `M01-F08` is not yet built. For now, the adjustment
  handler creates the entries but does not write an audit row. When M01-F08
  lands, add the audit hook.

- **`CreatedByUserId` population** — the handler reads this from
  `ICurrentUserService.UserId`. For system-generated entries (shift close,
  seeding), the calling handler must pass the appropriate user ID. This is
  the responsibility of the consuming feature, not this PR.
