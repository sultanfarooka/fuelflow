## Summary

M07-F08 makes Fuel Flow installable and offline-launchable for shift operations
on shared station tablets. It introduces `vite-plugin-pwa` (the tool already
named in `fuel-flow-web/CLAUDE.md`), a Workbox-precached **app shell** so the
chrome loads with no network, a complete **web app manifest** (name "Fuel Flow",
192/512/maskable icons, theme color) that satisfies "Add to Home Screen", and a
single **global offline banner** that surfaces lost connectivity and lets users
retry. Per the planning interview the service worker **auto-updates silently**,
only the **static app shell is cached** (no API/data caching, no offline write
queue), and offline detection is driven by `navigator.onLine` + `online`/
`offline` events **and** Axios network errors. Web push (R04) is out of scope
(v2).

## Acceptance criteria (the test plan)

`MODULES.md` carries **no** acceptance criteria for M07-F08 today. These four are
authored here and **added to the `MODULES.md` M07-F08 section** in the shipping
PR (ACs live in `MODULES.md` per `docs/CLAUDE.md`). They become Playwright test
names like `M07_F08_R02_ManifestInstallable`:

- **AC1 (R02)** — Given a production build served by the preview server, When the
  app loads in Chromium, Then a valid `manifest.webmanifest` is served with
  `name = "Fuel Flow"`, `display = "standalone"`, a `theme_color`, and icons
  including at least a 192px, a 512px, and a `purpose: "maskable"` entry; the
  manifest parses with no errors and the app meets the browser's installability
  criteria.
- **AC2 (R01)** — Given the app has been loaded once (service worker installed,
  shell precached), When the network is offline and the user reloads / relaunches,
  Then the app shell renders (no browser "you're offline" error page).
- **AC3 (R03)** — Given the app is running, When the network goes offline (or an
  API call fails with a network error), Then a single global offline retry banner
  appears on whatever screen the user is on, and data requests fail gracefully
  with no unhandled crash; When connectivity returns (`online` event), Then the
  banner auto-dismisses.
- **AC4 (R01)** — Given a new build is deployed (new precache revision), When an
  installed client next reloads / navigates, Then the new service worker takes
  over and the client runs the new version without a manual cache clear
  (`registerType: 'autoUpdate'`). _Primarily verified by config inspection + a
  manual two-build check; e2e coverage best-effort (SW update timing is flaky in
  Playwright)._

Per-requirement coverage:
- **R01** Service worker caches the app shell for offline launch → AC2, AC4.
- **R02** Web app manifest with icons + name supports "Add to Home Screen" → AC1.
- **R03** API calls fail gracefully when offline (retry banner, no queue) → AC3.
- **R04** Web push notifications → **Out of Scope (v2)** — excluded.

## Decisions (from planning interview)

1. **Tooling → `vite-plugin-pwa`.** The documented intended tool for this Vite +
   React project (`fuel-flow-web/CLAUDE.md`). Workbox under the hood, generates
   the SW + injects the precache manifest + wires registration. (KISS — no
   hand-rolled SW.)
2. **Update strategy → silent auto-update.** `registerType: 'autoUpdate'`. New SW
   activates and the page picks it up on next navigation/reload — no "new version"
   prompt UI. Best for shared station tablets that must never run a stale build.
3. **Caching scope → app shell only.** Workbox **precaches** the static build
   output (JS/CSS/fonts/icons/`index.html`) and uses `index.html` as the SPA
   navigation fallback. API responses are **not** cached. (YAGNI; no stale-data
   risk.)
4. **Offline UX → one global banner.** A single `<OfflineBanner />` driven by a
   shared `useOnlineStatus` hook (`navigator.onLine` + `online`/`offline`
   events), also raised by Axios network errors. Mounted once near the app root
   so it covers every screen; auto-dismisses when back online.
5. **Install identity → generated placeholders.** Author one source SVG mark and
   generate `pwa-192x192.png`, `pwa-512x512.png`, `maskable-icon-512x512.png`,
   `apple-touch-icon-180x180.png` (+ `favicon.ico`) into `public/`; manifest
   `name = "Fuel Flow"`, `theme_color` from `--primary`, `background_color` from
   `--background`.

## Implementation notes

**Phase 1.**
- **Theme hex resolved (open question closed).** `oklch(0.553 0.195 38.402)` →
  exact gamut-mapped sRGB **`#ca3500`** (computed with the OKLab→sRGB matrices);
  `oklch(1 0 0)` → `#ffffff`. Manifest `theme_color`/`background_color` and the
  `index.html` `<meta name="theme-color">` all use `#ca3500` / `#ffffff`.
- **Icons via `@vite-pwa/assets-generator` preset `minimal-2023`** — emitted
  `pwa-64x64/192x192/512x512`, `maskable-icon-512x512`, `apple-touch-icon-180x180`,
  `favicon.ico`. The 64px variant is extra (preset default) and harmless; the
  manifest lists 64/192/512 + maskable.
- **`includeAssets`** added so `favicon.ico`, `apple-touch-icon-180x180.png`, and
  `app-icon.svg` are precached alongside the build output.
- **Default `vite.svg` removed.** `index.html` previously referenced `/vite.svg`
  (the file existed in `public/`); the reference was replaced with the generated
  favicon/SVG/apple links and the now-unused `public/vite.svg` was deleted in
  Phase 2 so it stops being precached.

**Phase 2.**
- **App-shell precache verified at build time.** 20 precache entries incl.
  `index.html`, `assets/index-*.js`, `assets/index-*.css`, fonts, and icons;
  `grep "/api/" dist/sw.js` = 0 (no API caching). `navigateFallback: index.html`
  is present in the generated `sw.js`. SW registration + manifest link are
  auto-injected into the built `index.html`.
- The **offline-reload** and **auto-update** runtime checks (AC2/AC4) are
  exercised by the Playwright e2e (Phase 5) and the Step 4.5 MCP walk, not from
  the build — their boxes stay open until then.

**Phase 3.**
- **Decoupled network signal.** `lib/network-status.ts` exposes
  `emitNetworkError()` / `emitNetworkOk()` over plain `window` events so the
  Axios client never imports React or a store. `client.ts` emits an error on any
  response with no `error.response` (excluding `axios.isCancel`) and an "ok" on
  the next successful response; the 401-refresh / redirect path is unchanged.
- **i18n moved into Phase 3.** The plan put the `offline.*` keys in Phase 4, but
  a banner referencing missing keys would render raw keys in the Phase 3 commit,
  so the keys ship with the component. Phase 4 now only carries the scoped-doc
  update. Added an extra `offline.unreachable` (browser online but API down) on
  top of the planned title/body/retry.
- **Theme tokens.** The preset has no `--color-destructive-foreground`, so the
  banner uses `bg-destructive` + `text-white` (good contrast) rather than a
  non-existent foreground token; border is `border-black/10`. Logical utilities
  (`ms-`/`me-`) keep it RTL-correct.
- **Placement.** Mounted in `main.tsx` inside `QueryClientProvider` (so
  `useQueryClient` works for Retry) as a `fixed inset-x-0 top-0 z-[100]` bar.
