## Summary

Delivers physical per-tenant database routing and provisioning in one PR.
M14-F02 introduces `TenantDbContextAccessor` — a scoped service that resolves the
correct `AppDbContext` connection string once per HTTP request from the JWT `org_id`
claim, and refactors all 11 per-tenant repositories and `UnitOfWork` to use it.
M14-F03 introduces `TenantProvisioningService` which, during onboarding step 1,
creates a Postgres database (`tenant_<orgId>`), migrates it, inserts the
`Organization` row, and flips the control-plane `Tenant` status to `Active`.
After this PR every authenticated tenant request physically hits its own database.

## Acceptance criteria (the test plan)

- **AC1** — `TenantDbContextAccessor` resolves to the correct tenant DB for a
  request with a valid `org_id` JWT claim; queries hit the right Postgres database.
- **AC2** — Unauthenticated requests (no `org_id`) touching only control-plane
  repos complete without error; the accessor is never accessed.
- **AC3** — A request with an `org_id` that has no `Tenant` row returns HTTP 503.
- **AC4** — Completing onboarding step 1 creates `tenant_<orgId>` Postgres DB, a
  `Tenant` row (Status `Active`) in the control plane, and an `Organization` row in
  the tenant DB.
- **AC5** — JWT re-issued after onboarding step 1 carries `org_id`; subsequent
  wizard steps route to the tenant DB.
- **AC6** — Provisioning failure at any step leaves no orphaned DB and no
  `Tenant` row in the control plane. Response is HTTP 500.
- **AC7** — Full Playwright E2E: registration → phone OTP → login → onboarding
  wizard → station/tank/nozzle CRUD → logout → re-login passes end-to-end on a
  fresh local dev database.

### Phase 7 — TenantProvisioningService (Infrastructure)

- [ ] Add `FuelFlow.Infrastructure/Services/TenantProvisioningService.cs`
      implementing `ITenantProvisioningService`:
  1. Insert `Tenant` row (`Status = Provisioning`, `DatabaseName = "tenant_{orgId:N}"`)
     into `ControlPlaneDbContext`; save.
  2. Open a raw `NpgsqlConnection` using the base `DefaultConnection` string (the
     control-plane DB). Execute `CREATE DATABASE "tenant_{orgId:N}"` — this is a
     DDL statement that must run **outside** any transaction (auto-commit).
  3. Build `DbContextOptions<AppDbContext>` targeting the new DB (replace database
     name in connection string via `NpgsqlConnectionStringBuilder`). Create an
     `AppDbContext` directly (`new AppDbContext(options)`). Call
     `await ctx.Database.MigrateAsync(ct)`.
  4. Insert `Organization` row (`Id = orgId`, `Name = orgName`, `OwnerId = ownerId`,
     `CreatedAt = UtcNow`, `UpdatedAt = UtcNow`) via that context; save.
  5. Flip `Tenant.Status = Active`, `ProvisionedAt = UtcNow`; save control plane.
- [ ] Compensating actions wrapped in try/catch:
      - If step 2+ succeeded → `DROP DATABASE IF EXISTS "tenant_{orgId:N}"` on a
        separate `NpgsqlConnection`.
      - Always delete the `Tenant` row from `ControlPlaneDbContext` on any failure.
- [ ] Register `services.AddScoped<ITenantProvisioningService, TenantProvisioningService>()`
      in `DependencyInjection.cs`.
- [ ] **Acceptance:** Calling `ProvisionAsync` on a clean dev DB creates the
      database, runs all tenant migrations, and inserts the Organization row.
