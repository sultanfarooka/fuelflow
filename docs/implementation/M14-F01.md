## Summary

Split the single `AppDbContext` into two: a `ControlPlaneDbContext` carrying
Identity (`AppUser`, `AppRole`, etc.), the new `Tenant` registry,
`Subscription`, `SubscriptionPlans`, `OMC`, `OMCFuelTypes`, `FuelType`,
`PhoneVerification`, and `RefreshToken`; and an `AppDbContext` carrying
`Organization`, `Station`, `FuelTank`, `FuelNozzle`, `FuelPrices`,
`StationShift`, `ShiftAssignment`, `NozzleReadings`, `FuelTankReading`,
`DipChart`, `DipChartEntry`, `StationShiftConfig`, `BankAccount`, and
`UserStation`. Both contexts target the same physical Postgres database in
this PR — per-tenant physical isolation arrives in [M14-F02](../MODULES.md#m14-f02--tenant-registry--connection-resolution) and
[M14-F03](../MODULES.md#m14-f03--tenant-provisioning-service). Migration histories are rewritten as two fresh `Initial`
migrations (no production data exists; dev DBs are wiped). All cross-context
navigation properties are dropped — referential integrity for cross-context
references (`Organization.OwnerId`, `UserStation.UserId`,
`Subscription.OrganizationId`) becomes an app-layer concern enforced in
handlers. M14-F01 must be invisible to users; the existing M01-F09 and
M12-F01/F02 user-visible behaviors must work exactly as before.

## Open questions

- **Custom FuelType per tenant?** Today the `FuelType` entity has an `IsCustom` flag, suggesting per-tenant custom fuel types were anticipated. M14-F01 puts FuelType in the control plane (platform-wide). If custom per-tenant fuel types become a real ask later, FuelType moves to tenant-side and the seed strategy changes. **Decision for F01:** platform-wide; revisit when a real custom-fuel-type ask arrives.
- **`OnboardingCommandHandler` saga shape (deferred).** F01 keeps the existing `TransactionScope` because both contexts share a physical DB. F03 will require the real saga (compensate by deleting the control-plane Tenant row + DROPping the tenant DB if any step fails). Mark with `// TODO M14-F03 — replace with explicit saga + compensation` at the call site.
- **Cross-DB cascade deletes (deferred).** Deleting an `AppUser` today cascades to their `UserStation` rows via FK. After F01 the FK is gone; app-layer enforcement is required. F01 ships without this (no Identity user deletion happens in the codebase today). Mark with `// TODO M14-F05 — enforce UserStation cleanup when AppUser is deleted` in `AppUserRepository.DeleteAsync` (if it exists).
