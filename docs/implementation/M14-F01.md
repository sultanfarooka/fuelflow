## Summary

Split the single `AppDbContext` into two: a `ControlPlaneDbContext` carrying
Identity (`AppUser`, `AppRole`, etc.), the new `Tenant` registry,
`Subscription`, `SubscriptionPlans`, `OMC`, `OMCFuelTypes`, `FuelType`,
`PhoneVerification`, and `RefreshToken`; and an `AppDbContext` carrying
`Organization`, `Station`, `FuelTank`, `FuelNozzle`, `FuelPrices`,
`StationShift`, `ShiftAssignment`, `NozzleReadings`, `FuelTankReading`,
`DipChart`, `DipChartEntry`, `StationShiftConfig`, `BankAccount`, and
`UserStation`. Both contexts target the same physical Postgres database in
this PR ‚Äî per-tenant physical isolation arrives in [M14-F02](../MODULES.md#m14-f02--tenant-registry--connection-resolution) and
[M14-F03](../MODULES.md#m14-f03--tenant-provisioning-service). Migration histories are rewritten as two fresh `Initial`
migrations (no production data exists; dev DBs are wiped). All cross-context
navigation properties are dropped ‚Äî referential integrity for cross-context
references (`Organization.OwnerId`, `UserStation.UserId`,
`Subscription.OrganizationId`) becomes an app-layer concern enforced in
handlers. M14-F01 must be invisible to users; the existing M01-F09 and
M12-F01/F02 user-visible behaviors must work exactly as before.

## Implementation notes

The smoke walk approach in the M14-F01 spec deliberately differs from M12-F02's pattern. M14-F01 doesn't introduce any new user-visible behaviour, so the spec verifies the *invariants* M14-F01 must preserve (backend boots, Identity stack works, dashboard hydrates) rather than walking the full register ‚Üí OTP ‚Üí wizard journey. The full journey is M12-F01/F02's spec to own; M14-F01 leaning on that spec's R02 passing is the regression signal we needed.

Phase 4's model-validation issues (FuelType.Station nav, three `HasOne<AppUser>()` blocks, missing ShiftAssignmentConfiguration) were caught by `dotnet ef migrations add` design-time validation ‚Äî not at runtime. That's a useful signal: when you make a structural change to a DbContext model, run `dotnet ef migrations add Empty --context X` against both contexts as a smoke test before committing. If it fails, the model has a problem you should fix BEFORE any runtime work.
