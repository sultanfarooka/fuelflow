## Summary

Post-onboarding self-service management of a station's nozzles for Owner + Manager, rendered as the fourth child of the Station Configuration hub. Replaces the `<UnderDevelopment />` placeholder. Users see a list of nozzles at the station with number + linked tank (+ implied fuel type) + active/inactive state + shift-assignment count; can **add** a nozzle (number + tank required), **edit** number / tank / active state (with extra confirmation when reassigning the tank), **toggle active/inactive via a Switch**, and **delete** with a structured 409 reference-guard listing blocking shift-assignments. Implements R01 (CRUD UI) and confirms R02 (uniqueness). **Out of scope:** mid-shift swap, nozzle calibration / totalizer init (`M03`), reading history.

## Key model facts (from code review during planning)

- **`FuelNozzle` is per-tenant** (`AppDbContext`, table `fuel_nozzles`). Columns: `Id`, `NozzleNumber` (`varchar(20)`, free text), `TankId` (FK → `FuelTank`), `StationId`, **`IsActive` (bool, default true)**. - **The `IsActive` flag is the key differentiator from M08-F02.** Tanks were hard-delete-only because they had no `IsActive`. Nozzles get both Switch (soft) AND Delete (hard, with ref-guard) — the operational "nozzle under maintenance" state matters here, and matching M08-F08's pattern uses what the schema already provides.
- **Backend partially built:**

  - Repo: `IFuelNozzleRepository.GetByStationIdAsync / GetByIdAsync / AddAsync / DeleteAsync` ✓; **missing**: `UpdateAsync` (or `Update` for tracked entities), reference-count helper
  - Controller: `[Authorize(Roles="Owner,Manager")] [Route("api/v1/stations/{stationId:guid}/fuel-nozzles")]` GET / POST / DELETE ✓; **missing**: PUT `/{nozzleId}` (rename + reassign), PATCH `/{nozzleId}/status` (toggle)

## Decisions (resolved during planning interview)

1. **Both Switch (soft-deactivate) AND Delete (hard, 409 ref-guard).** `IsActive` already exists, so nozzle "under maintenance" is a first-class state. Switch handling mirrors `FuelTypeToggle` from M08-F08 — flipping ON activates immediately; flipping OFF opens a confirm dialog (so the switch snaps back if cancelled, and so the user sees the implication). Delete is permanent + structured 409 with `ShiftAssignment` count via the same preflight pattern as M08-F02.
2. **Tank reassignment on Edit requires second-click confirm.** When `tankId` differs from the original, render a destructive-tone Alert ("Existing shift records keep the old tank reference for reports. Confirm carefully.") and the Save button becomes `variant="destructive"` with label "Tap again to confirm". Same UX as `FuelTanksPanel`'s fuel-type-reassignment flow.
3. **`NozzleNumber` stays free text, 1–20 chars, per-station unique.** No regex. Backend uniqueness (R02) is already Done.
4. **Audit + structured 409 mirror sibling panels.** Serilog `AUDIT FuelNozzle.Create / Update / SetActive / Delete` lines + `AUDIT FuelNozzle.Delete.Blocked` + `AUDIT FuelNozzle.Deactivate.Blocked` (only if we add a deactivate guard — see Open questions). New `DeleteFuelNozzleResponse` DTO carries `Blocked` + `BlockingReferences` exactly like `DeleteFuelTankResponse` and `SetFuelTypeActiveResponse`.
5. **Visual chrome inherits from M08-F08 + M06-F01 + M08-F02 by reuse** — fourth adopter of `<ConfigPanelCard>` + shared `<DataTable>` + extracted badges. No new shell extractions required this round.
6. **Closes the Sellable loop end-to-end.** A fuel type with an active price (M06-F01) + a tank (M08-F02) flips to `Sellable` (M08-F08). Adding nozzles (this PR) lets the operational side actually sell. No code change needed for the badge — already wired.

## Open questions

- **`IShiftAssignmentRepository` may not have `CountByNozzleIdAsync`.** If a repo method doesn't exist, two paths: (a) extend the repo with one method (clean), or (b) load `GetByStationIdAsync` for shift assignments and group in-memory (matches M08-F02's nozzle-count pattern but assumes the repo method exists). Confirm during Phase 1 — proposed: extend the repo with a count method since the assignments table can grow large.
- **Should deactivating a nozzle with an open shift be blocked?** `StationShift` doesn't directly reference nozzles (assignments do), and we currently don't preflight shifts. Proposed: no — soft-deactivate is harmless; the assignment row keeps its nozzleId reference for reports. Revisit when M04 shifts ship.
- **Tank `<Select>` — show only tanks that have a fuel type with an active price?** Tighter UX (Sellable tank only), but might be surprising. Proposed: no — show all tanks; assigning a nozzle to a not-yet-sellable tank just means the nozzle won't dispense until price + fuel-type are set. Matches the looseness of the Fuel Types and Tanks panels.
- **`UpdateFuelNozzleRequest.IsActive` field — include it in the PUT payload or rely on the PATCH `/status` endpoint?** Proposed: PUT does NOT touch `IsActive` (Edit dialog has no Active toggle; the Switch lives in the table). Keeps the two endpoints' concerns clean — same separation M08-F08 already uses (rename via PUT, activate/deactivate via PATCH `/status`).

## Implementation notes

- **Open-question defaults applied without prompt** (Auto Mode): Q1 (count via repo method) — yes, added `CountByNozzleIdAsync` + batch `CountByNozzleIdsAsync`. Q2 (block deactivate on open shift) — no, soft-deactivate is harmless. Q3 (Select shows only sellable tanks) — no, show all active tanks. Q4 (PUT touches IsActive) — no, PUT updates NozzleNumber + TankId only; IsActive is via the PATCH `/status` endpoint exclusively.
- **Existing Create handler enforces per-TANK uniqueness, not per-station.** MODULES.md R02 says "Nozzle number uniqueness validated per station" but the existing `CreateFuelNozzleCommandHandler` scopes uniqueness to `(stationId, tankId, nozzleNumber)`. The new `UpdateFuelNozzleCommandHandler` matches the existing scope for behaviour consistency. The R02 spec-vs-code mismatch is out of scope for this PR (would need a stakeholder call on the intent); I've documented it here so a future audit can decide whether to tighten the rule or update R02.
- **`SetFuelNozzleActiveCommand` is idempotent.** Setting active=true on an already-active nozzle (or active=false on already-inactive) returns success without writing or emitting an audit line. Prevents log noise when the UI's optimistic Switch flips back-and-forth.
