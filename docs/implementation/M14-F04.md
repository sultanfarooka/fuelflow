## Summary

Purely frontend. Adds a full-screen "Provisioning your workspace…" overlay
(`ProvisioningOverlay`) that replaces the wizard while step 1's `POST /onboarding`
is in flight (~5–30s). If provisioning fails (HTTP 500) the overlay is dismissed,
the form re-enables, and a Sonner toast + inline `<Alert>` prompt the user to
retry. No changes to wizard steps 2–9 — they already read from the auth store,
which is updated by `setAuthState(auth)` on step 1 success.

## Acceptance criteria (the test plan)

- **AC1** — After step 1 submission the full-screen overlay replaces the wizard
  content; no wizard interaction is possible while it is visible; pressing the
  browser Back button triggers a `beforeunload` warning.
- **AC2** — When `POST /onboarding` succeeds, the overlay disappears and the
  wizard advances to step 2; the auth store is updated with the new JWT
  carrying `org_id`.
- **AC3** — When `POST /onboarding` returns HTTP 500, the overlay is dismissed,
  the form re-enables, a Sonner error toast fires, and the inline `<Alert>`
  shows the error; the user can retry without a page reload.
- **AC4** — Playwright spec `M14-F04.spec.ts` passes end-to-end: registration →
  phone OTP → login → onboarding (provisioning overlay observed in step 1) →
  wizard completion → station/tank/nozzle CRUD.
